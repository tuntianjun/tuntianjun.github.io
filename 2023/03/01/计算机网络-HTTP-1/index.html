计算机网络--HTTP(1)

发表于2023-03-01|更新于2023-03-01
|阅读量:

基本概念

HTTP的定义

  • HTTP即超文本传输协议，主要由三部分构成，按照范围划分：协议->传输->超文本
  • 协议：计算机之间进行通信以及控制和错误处理的行为规范
  • 传输：计算机之间进行数据传输的约定（HTTP是双向协议）
  • 超文本：文字、图片、音视频等，关键在于超链接（能从某个超文本跳转到另一个超文本）
  • 综上，HTTP是一个在计算机两点之间传输文字、图像、音视频等超文本数据的约定和规范

问：HTTP是用于从互联网服务器传输超文本到本地浏览器的协议，这种说法正确吗？
答：不正确，也可以在两台服务器之间传输超文本数据

HTTP常见状态码

  • 1xx 类状态码属于提示信息，是协议处理中的一种中间状态，实际用到的比较少
  • 2xx 类状态码表示服务器成功处理了客户端的请求，也是我们最愿意看到的状态
    • 「200 OK」是最常见的成功状态码，表示一切正常。如果是非 HEAD 请求，服务器返回的响应头都会有 body 数据。
    • 「204 No Content」也是常见的成功状态码，与 200 OK 基本相同，但响应头没有 body 数据。
    • 「206 Partial Content」是应用于 HTTP 分块下载或断点续传，表示响应返回的 body 数据并不是资源的全部，而是其中的一部分，也是服务器处理成功的状态。
  • 3xx 类状态码表示客户端请求的资源发生了变动，需要客户端用新的 URL 重新发送请求获取资源，也就是重定向
    • 「301 Moved Permanently」表示永久重定向，说明请求的资源已经不存在了，需改用新的 URL 再次访问。
    • 「302 Found」表示临时重定向，说明请求的资源还在，但暂时需要用另一个 URL 来访问。
    • 301 和 302 都会在响应头里使用字段 Location，指明后续要跳转的 URL，浏览器会自动重定向新的 URL。
    • 「304 Not Modified」不具有跳转的含义，表示资源未修改，重定向已存在的缓冲文件，也称缓存重定向，也就是告诉客户端可以继续使用缓存资源，用于缓存控制。
  • 4xx 类状态码表示客户端发送的报文有误，服务器无法处理，也就是错误码的含义。
    • 「400 Bad Request」表示客户端请求的报文有错误，但只是个笼统的错误。
    • 「403 Forbidden」表示服务器禁止访问资源，并不是客户端的请求出错。
    • 「404 Not Found」表示请求的资源在服务器上不存在或未找到，所以无法提供给客户端。
  • 5xx 类状态码表示客户端请求报文正确，但是服务器处理时内部发生了错误，属于服务器端的错误码。
    • 「500 Internal Server Error」与 400 类型，是个笼统通用的错误码，服务器发生了什么错误，我们并不知道。
    • 「501 Not Implemented」表示客户端请求的功能还不支持，类似“即将开业，敬请期待”的意思。
    • 「502 Bad Gateway」通常是服务器作为网关或代理时返回的错误码，表示服务器自身工作正常，访问后端服务器发生了错误。
    • 「503 Service Unavailable」表示服务器当前很忙，暂时无法响应客户端，类似“网络服务正忙，请稍后重试”的意思。

HTTP常见字段

  • Host字段：用来指定服务器域名
  • Content-Length字段：表示服务器回应的数据长度（和HTTP协议中的回车、换行符类似，都可以用来解决“粘包”问题，具体内容见TCP部分）
  • Connection字段：常用于客户端要求服务器使用HTTP长连接机制，只要一端没有明确提出断开请求，就一直保持TCP连接，客户端发送的另一个请求，仍然会使用相同的连接

    TCP中的Keepalive和HTTP中的Connection:Keep-Alive是不一致的，详见TCP部分

  • Content-Type字段：用于服务器向客户端回应本次数据的格式，例如：Content-Type: text/html; Charset=utf-8，表明发送的是网页，使用UTF=8编码；客户端可以使用Accept字段告知服务器自己可以接收那些类型的数据
  • Content-Encoding字段：服务器向客户端回应使用的数据压缩方式，客户端可用Accept-Encoding字段告知服务器端自己可以接收何种压缩方式

GET和POST

  • GET：从服务器获取指定的资源，GET请求参数（只允许ASCII码字符）是写在URL中的，尽管HTTP协议对于URL的长度没有限制，但浏览器对于URL的长度会做出限制
  • POST：根据请求的数据载荷（请求报文的数据主体内容）来对指定的资源做出处理，POST请求携带数据的位置一般在报文主体部分，主体中的数据没有格式要求，只要客户端和服务器协商即可，浏览器对报文主体大小不做限制

HTTP中的安全和幂等

  • 安全：请求方法不会破坏服务器上的资源
  • 幂等：多次执行相同操作，得到的结果是相同的

问：GET和POST都是安全和幂等的吗？
答：从RFC规范文档来看：
GET方法时安全且幂等的，它只是只读操作，不会对服务器上的数据造成损害且每次结果都是相同的；GET是可以被缓存的
POST是新增或提交操作，会修改服务器资源，是不安全的。并且，多次提交得到的结果是不一样的，所以不是幂等的；POST大部分是不可悲缓存的

问：GET请求可以带body吗？
答：RFC文档没有明确规定不能带，但FET是用来请求资源的，没有必要带body；同时GET请求的URL请求参数也不是GET独有的，POST的URL中也可以有请求参数

HTTP缓存技术

  • 如果需要多次请求同样的数据，为避免重复等待服务器的响应，可以将请求-响应的数据缓存在本地，下次遇到同样的请求就直接读取本地缓存数据
  • HTTP缓存分为两类：强制缓存和协商缓存

HTTP强制缓存方式

  • 强制缓存：决定权在浏览器手中，只要浏览器判断缓存没有过期，就直接使用浏览器本地缓存
  • 在HTTP响应头部中，有以下两个字段实现强制缓存，均表示缓存的有效期：
    • Cache-Control：相对时间
    • Expires：绝对时间
    • 同时有上面两个字段，则Cache-Control优先级更高
  • Cache-Control实现强制缓存的流程：
    • 浏览器第一次请求服务器资源是，服务器返回该资源并在响应头部加上Cache-Control
    • 浏览器再次请求，会先通过请求资源的时间与Cache-Control中设置的过期时间比较，判断资源是否过期
    • 没过期就直接使用本地缓存，过期就重复第一步的步骤

HTTP协商缓存

  • 协商缓存：客户端和服务器端进行协商，根据协商结果判断是否使用本地缓存，未能命中强制缓存，才能发起协商缓存
  • 协商缓存同样有两种请求头部实现，但这两种头部都需要配合Cache-Control使用：
    • If-Modified-Since和Last-Modified字段：
      • If-Modified-Since：若资源过期，响应头中有Last-Modified，则再次请求时服务器将Last-Modified与If-Modified-Since对比。如果Last-Modified时间更新，说明资源被修改过，则返回最新资源，返回200状态码；反之，说明该资源没有被修改过，返回304状态码，继续使用本地缓存。
      • Last-Modified：表示响应资源的最后修改时间
    • If-None-Match和ETag字段：
      • If-None-Match：若资源过期，响应头中有ETag，则再次请求时服务器将If-None-Match的值设置为ETag的值。服务器收到请求后进行对比，如果资源没有变化返回304状态码，变化则返回200状态码。
      • ETag：唯一标识响应资源

问：Last-Modified和ETag同时出现，后者优先级更高，为什么？
答：第一，尽管没有修改文件其最后修改时间也可能改变；第二，文件的修改可能在秒级以内完成，If-Modified-Since检查的粒度是秒级，使用ETag就能保证客户端在1秒内能刷新多次；第三，有些服务器无法准确获取文件的最后修改时间

  • ETag实现协商缓存的过程：
    • 首次请求服务器资源，服务器返回资源时会在响应头加上ETag标识
    • 再次请求该资源时，会先检查强制缓存是否过期：
      • 没有过期就使用本地缓存
      • 过期就加上If-None-Match字段，并设为ETag的值
    • 服务器收到请求后，将If-None-Match的值与请求的资源ETag值对比：
      • 若相等则返回304
      • 若不相等则返回200和新资源，更新ETag的值

HTTP特性

本部分先由HTTP/1.1介绍HTTP的特性，2、3的改进会在后面介绍

HTTP/1.1的优点

  • 简单：报文格式简单，易于理解
  • 灵活、易于拓展：各类HTTP协议的报文组成没有被固定死，允许开发人员自定义和扩充；工作在应用层，下层可随意变化
  • 应用广泛和跨平台：从PC到移动设备，有跨平台的优势

HTTP/1.1的缺点

  • 无状态：
    • 优点：不记录HTTP状态，无需额外资源记录状态信息，减轻服务器负担
    • 缺点：在完成有关联性的操作时会很麻烦，如：网购时，登录、加购物车、下单、结算、支付，每一步都要询问用户身份信息。解决方法有很多，常见的就是cookie技术，服务器在请求和响应报文中添加cookie信息来记录客户端的状态。
  • 明文传输：
    • 优点：传输中的信息方便阅读，调试工作遍历
    • 缺点：信息暴露，易被窃取
  • 不会验证通行方的身份，可能遭遇伪装
  • 无法证明报文的完整性，可能遭到篡改

HTTP/1.1的性能

  • HTTP协议基于TCP/IP协议，使用请求-应答的通信方式

  • 长连接： HTTP/1.0每次进行发起请求都要建立一个新的TCP连接，通信开销太大。HTTP/1.1为解决该问题，提出长连接通信，即只要任意一端没有明确提出断开连接，就保持TCP连接（如果长时间没有数据交互，服务器端会主动断开连接）。

  • 管道网络传输： 由于长连接的通信，管道网络传输也成为可能，即同一个TCP连接中，客户端可以发起多个请求（只要第一个请求发出，不必等待回应就可发送第二个请求），减少整体的响应时间。

    实际上，该技术并非默认开启，当时浏览器基本没有支持该技术

  • 队头阻塞： 当某个请求长时间没有得到响应时，后续的请求无法发送，这就造成了请求的队头阻塞

    管道网络传输解决了请求的队头阻塞，但是没有解决响应的队头阻塞，

HTTP与HTTPS

二者区别

  • 前面提到了HTTP是明文传输的，有诸多数据不安全的问题，而HTTPS则通过在TCP和HTTP网络层之间添加SSL/TLS安全协议，将报文加密传输
  • HTTPS在TCP三次握手后，还要进行SSL/TLS握手，才能进行加密传输
  • HTTP默认端口80，HTTPS默认端口443
  • HTTPS需要想CA（整数权威机构）申请数字证书，保证服务器的身份可信

HTTPS如何解决不安全问题

  • 混合加密：实现信息的机密性，解决被窃听的问题
    • 通信前使用非对称加密，交换会话秘钥，非对称加密使用公钥和私钥，公钥任意分发，私钥保密，运算速度慢
    • 通信中使用对称加密来加密明文数据，对称加密使用一个秘钥，运算速度快
  • 摘要算法：实现信息的完整性
    • 对传输内容计算一个哈希值，和内容一起发送
    • 接收方收到后，也对内容计算哈希值，若和发送方一致，则表明内容没有被修改
    • 如果发送方的数据和哈希值在发送过程中被人调换了，如何解决？
    • 用非对称加密解决：
      • 公钥加密，私钥解密：保证传输内容的安全，因为只有持有私钥的人才能解密传输内容
      • 私钥加密，公钥解密：保证传输内容不会被人冒充或调换，因为私钥不可泄露，如果能用公钥解密，则说明该传输内容的发送人是正确的，传输内容没有被冒充，这种方式也就是常说的数字签名算法（对内容的哈希值进行私钥加密）
  • 数字证书：
    • 假如公钥被伪造，在私钥加密后传输内容，可以用伪造的公钥对应的私钥来加密一段假内容冒充原本加密传输的内容，这样的情况如何解决？
    • 下面是整个流程：
      • 将公钥内容拿到数字证书认证机构（CA）注册，CA将持有者的公钥、用途有效期等打包，对这个数据包计算hash值，用他们的私钥对该hash值加密，并添加到问价证书上形成数字证书
      • 客户端对拿到的证书计算hash值，浏览器用公钥解密证书内容得到CA计算的hash值，若两个hash值相同则说明是可信任的证书

关于证书的信任链： 许多证书并非由根证书直接签发，而是根证书信任某个下级证书，而该下级证书又信任另一个下级证书，后面的这个下级证书是实际为我们签发证书的CA。最开始客户端只信任根证书，但由操作从而一步一步形成信任链，这样可以保证根证书的安全。

SSL/TLS协议

  • TLS握手设计4次通信，使用不同的秘钥交换算法，常用的：RSA算法和ECDHE算法
  • 具体流程

HTTPS应用数据如何保证完整性

  • TSL协议分为握手协议和记录协议：
    • 握手协议负责加密，保护数据
    • 记录协议负责验证数据完整性和来源，具体如下：
      • 首先消息被分割为多个短片段，对每个片段进行压缩
      • 压缩的片段加上消息认证码（有哈希算法生成的MAC值），保证完整性
      • 片段和消息认证码通过对称加密，添加报头形成报文数据

HTTPS一定安全吗

  • 如果有假基站起到转发所有信息的作用，该假基站是否就获得了全部的信息？

  • 简单来说就是客户端向服务端发起请求时，被中间服务端拦截，中间服务端和客户端进行握手，又和真正的服务端握手，中间服务端因此能够查看客户端和真正服务端之间的请求和响应数据

  • 但是，发生这种情况有前提，

    • 一种是用户需要接受中间服务端的证书，这个证书是可以 被客户端浏览器识别为非法证书，所以如果用户执意信任该证书，上述情况是会发生的
    • 另一种是客户端由于病毒被植入了中间客户端的证书
    • 这两种都是由于客户端的问题而导致的信息泄露，和HTTPS没有太大的关系

  • 抓包工具能够截取HTTPS的数据，原理和上面的类似，抓包工具充当中间人，并且抓包工具会在客户端系统中导入自己生成的证书，给自己认证

  • 如何避免被人抓数据：

    • 不信任未知证书，避免病毒影响
    • 使用HTTPS双向认证，前面能抓取数据是因为服务端不会验证客户端的身份（服务端将中间人也当做客户端看），使用双向认证使得客户端与服务端之间会互相验明身份，解决上述问题

HTTP/1.1、HTTP/2、HTTP/3演变

1.1相较于1.0的改进

  • 使用长连接
  • 支持管道网络运输
  • 不足：
    • 请求和响应头部为压缩就发送，导致只能压缩报文主体部分
    • 首部冗余，造成网络资源浪费
    • 管道网络运输解决了请求的队头阻塞，没解决响应的队头阻塞
    • 没有请求优先级控制
    • 请求只能从客户端到服务端，服务器只能被动响应

2相较于1.1的优化

  • 2基于HTTPS，更安全

  • 头部进行了压缩，多个请求头部相似，会消除重复的部分（HPACK算法）。两端同时维护头部信息表，生成索引号，相同就发送索引号

  • 使用二进制格式，头部和数据都采用二进制，统称为帧，增加了传输效率

  • 并发传输，引入stream，多个stream复用一条TCP连接，多个消息通过stream ID来区分，解决了1.1中必须等上一个请求响应完后才能处理下一个响应的缺点

  • 服务器推送，客户端和服务端双方都能stream，客户端的stream ID是技术号，服务端是偶数号

  • HTTP2的缺点：

    • 解决了HTTP层面的队头阻塞，没解决TCP层面的队头阻塞。TCP传输字节流，当收到的字节数据不完整、不连续，数据会一致在内核缓冲区中等待（一旦丢包，所有请求都要等待丢包重传）

HTTP3的优化

  • 将TCP改为基于UDP的QUIC协议，实现类似TCP的可靠传输，解决阻塞，具体如下：
    • 无队头阻塞，当某个stream丢包，只阻塞这个stream，避免队头阻塞
    • 更快的连接建立
    • 连接迁移
