TCP/IP网络模型结构

![[Pasted image 20230227195800.png]]

1.应用层

  • 网络层位于最上层，用于处理进程间的通信（浏览器进程，邮件进程等）
  • 应用层的协议有：HTTP、FTP、Telnet、DNS、SMTP等
  • 引用层工作在操作系统中的用户态，传输层及以下工作在内核态

2.运输层

  • 运输层为应用层提供服务，有TCP和UDP两个传输协议
  • TCP 的全称叫传输控制协议（Transmission Control Protocol），是可靠、有连接的，有流量控制、超时重传、拥塞控制等
  • UDP是不可靠、无连接的，只发送数据包
  • 运输层在将数据传输给应用时，需要用编号将应用区分开，该编号就是端口

3. 网络层

  • 网际层最常用的协议是IP协议
  • IPv4地址的划分：分类编址、划分子网、无分类编址
  • 若将IP地址划分为：网络号+主机号，则网络号表示该地址属于哪个子网，主机号表示该地址属于某子网下的哪个主机
  • 将上述情况的IP地址和子网掩码做按位与得到网络号，与取反后的子网掩码做按位与得到主机号

4.网络接口层

  • 网络接口层通过MAC地址（可通过ARP协议获取）等信息判断数据发送的源地址和目的地址

5.数据的封装

浏览器访问网址究竟做了什么

从浏览器访问网址中把握计算机网络信息传输的整体流程

1.客户端浏览器进程

  • 首先，客户端浏览器进程对用户输入的URL进行解析，转换为待发送的HTTP请求报文
    ![[3.png]]

    问：若访问的URL不包含（a）中的蓝色部分，则请求的是什么？
    答：请求的是根目录下实现设置的默认文件：index.html或default.html等

2.DNS域名解析

  • URL并不是服务器端的实际地址，还需要通过DNS对服务器域名进行解析，得到对应的IP地址
  • 比如，在使用ping命令时，实际上ping的是IP地址
    ![[Pasted image 20230227204705.png]]
  • 域名解析的流程：
    • 客户端发送DNS请求，询问网址的IP，该请求发送给本地DNS服务器
    • 本地DNS服务器接收请求后，如果找到直接返回IP；
    • 如果找不到，则去询问根域名服务器，根域名服务器告知本地服务器应该去某个顶级域名服务器查找
    • 本地域名服务器又去询问顶级域名服务器，顶级域名服务器告知应该去权限域名服务器查找
    • 本地服务器在权限域名服务器处找到IP地址
    • 注：上述过程为迭代查找方式，更常用
      ![[Pasted image 20230227205436.png]]

问：每次解析域名都要这么多步吗？
答：浏览器、操作系统、hosts文件会先查找自己的缓存，没有才会询问本地域名服务器，本地域名服务器中也会将用户最近访问的域名数据放入高速缓存。

3.协议栈

  • 获得IP地址后，操作系统中的协议栈就会接手后面的工作
    ![[4.png]]
  • 浏览器调用Socket库来委托协议栈工作，协议栈包含：TCP、UDP和IP协议
  • 其中IP协议由ICMP和ARP组成，ICMP用于告知网络包传送过程中产生的错误以及各种控制信息，ARP根据IP地址查询相应的以太网MAC地址
  • 下面的驱动和硬件负责实际的接收和发送操作

4.TCP协议

  • 将得到的HTTP报文内容添加一个TCP首部，再交给网络层处理

  • TCP报文的首部格式如下：
    ![[Pasted image 20230227211505.png]]

  • 源端口和目的端口：指示发送和接收报文的应用

  • 序号：解决报文乱序

  • 确认号：解决丢包问题，用来确认对方是否收到报文

  • 状态位：URG=1，紧急指针才能用；ACK=1，建立连接；PSH=1，推送状态；RST=1，重置连接；FIN=1，结束连接；

  • 窗口：流量控制

  • 三次握手建立连接：
    ![[Pasted image 20230227212455.png]]

    • 开始时，客户端和服务端都处于关闭状态，服务端监听某端口，处于监听状态
    • 客户端发送连接请求，处于同步已发送状态
    • 服务端发送针对连接请求的确认报文，处于同步已接收状态
    • 客户端发送普通确认报文（没有SYN），处于连接已建立状态
    • 服务端接收到确认报文后，处于连接已建立状态

问：为什么要进行第三次握手？
答：若客户端发起连接请求的报文由于网络延时，未能即使发送，在触发超时重传后，客户端会再次发送连接请求。在一次完整的连接关闭后，该延时请求才到达，此时服务端误以为客户端要进行新的连接，因此发送确认请求报文，但客户端实际并未发送新的连接请求，因此不回应，从而导致网络资源的浪费。
![[Pasted image 20230227213406.png]]

  • 四次挥手断开连接：
    ![[Pasted image 20230227214412.png]]
    • 客户端发送连接释放请求，处于终止等待1状态
    • 服务端发送普通确认请求作为回应，处于关闭等待状态，此时客户端处于终止等待2状态，该状态下服务端仍可以进行数据传输，而客户端只能接收
    • 服务端数据传输结束后，发送连接释放请求，处于最后确认状态
    • 客户端发送普通确认请求作为回应，处于时间等待状态
    • 服务端收到确认，关闭
    • 客户端经过2倍最长报文寿命才会关闭

问：客户端为什么要进行等待再关闭？
答：若第四次挥手的确认报文丢失了，服务端的连接释放请求得不到回应，会进行多次超时重传，无法释放连接
![[Pasted image 20230227215335.png]]

  • 网络层将得到的数据再封装一个IP报头，交给下一层
  • IPv4报文首部格式如下，各部分具体含义不在此赘述：
    ![[Pasted image 20230227215833.png]]

问：当客户端有多个网卡，即有多个源IP地址，则应该选择哪一个？
答：根据路由表提供的信息，将目的地址和子网掩码分别做与运算，结果匹配成功的则是源IP地址，若都无法匹配，则自动匹配默认网关并将包发送给路由器

6.MAC地址

  • 数据链路层为封装IP首部的数据添加MAC地址以及帧首和帧尾，交由物理层处理

    问：MAC地址的发送方和接收方如何确定？
    答：发送方的 MAC 地址获取就比较简单了，MAC 地址是在网卡生产时写入到 ROM 里的，只要将这个值读取出来写入到 MAC 头部就可以了。
    接收方的 MAC 地址就有点复杂了，只要告诉以太网对方的 MAC 的地址，以太网就会帮我们把包发送过去，那么很显然这里应该填写对方的 MAC 地址。
    所以先得搞清楚应该把包发给谁，这个只要查一下路由表就知道了。

问：如何通过IP地址找到对应的MAC地址？（上一问中查找路由表的过程）
答：ARP地址解析协议，ARP会对以太网中的所有设备进行广播，询问对应IP的设备的MAC地址

问：每次都要广播吗？
答：ARP会在内存中有一块缓存空间，用来存储最近的查询结果；在执行ARP时，先在缓存中查找，找不到才会进行广播

  • 帧首和帧尾由网卡进行，帧首添加起始分界符来表示包的起始位置，帧尾添加帧检验序列（FCS）用来进行差错检验

7.物理层：交换机和路由器

8.总结

  • 数据通过电信号传输给服务器端
  • 服务器端通过相反的顺序一层一层解开各个首部，最终达到服务区的浏览器进程
  • 服务器浏览器进程将客户端请求的内容在由上到下封装后发送给客户端
  • 客户端再次进过一层一层解析，最终将网页内容呈现出来
